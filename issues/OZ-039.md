# OZ-039: String dedup counter collision across methods in same class

## Context

Discovered during test coverage work (PR 7: foundation-behavior). When multiple
methods in the same class use different `@"..."` string literals, the emitted
static string constants collide because the dedup counter resets per method.

## Input

```objc
@implementation Foo
- (const char *)getHello { return [@"hello" cStr]; }
- (const char *)getBye   { return [@"bye" cStr]; }
@end
```

## Observed

```c
static struct OZString _oz_str_0 = {{OZ_CLASS_OZString, 2147483647}, 5, 0, "hello"};
static struct OZString _oz_str_0 = {{OZ_CLASS_OZString, 2147483647}, 3, 0, "bye"};
```

GCC error: `redefinition of '_oz_str_0'`

## Expected

Each string constant should get a unique name across the entire translation unit:

```c
static struct OZString _oz_str_0 = {{OZ_CLASS_OZString, 2147483647}, 5, 0, "hello"};
static struct OZString _oz_str_1 = {{OZ_CLASS_OZString, 2147483647}, 3, 0, "bye"};
```

## Root Cause

The `_string_dedup` dict and counter in `_EmitCtx` are reset per method in
`_build_impl_context` (context.py:230-235). The counter should persist across
methods within the same class, or be scoped to the file/TU level.

## Impact

Blocks Foundation behavior tests for any class using different string literals
in separate methods. Affects OZDictionary tests (keys are strings) and any
multi-method class with varied `@"..."` usage.
