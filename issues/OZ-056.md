## Context

`+initialize` is currently a regular class method that must be called manually
(see `samples/arc_demo/src/main.m` line 111). Apple's ObjC runtime calls
`+initialize` automatically before the first message to a class. Objective-Z
should call it automatically before the application runs.

## Input

```objc
@implementation AppConfig
+ (void)initialize {
    _sharedConfig = [[AppConfig alloc] init];
}
@end

int main(void) {
    [AppConfig initialize];  /* manual call — should be automatic */
}
```

## Observed

Developer must manually call `[Class initialize]`. No automatic invocation.

## Expected

Three changes:

1. **Nil-guard inside +initialize** — generated code checks if singleton is
   already non-nil before allocating (idempotent safety).
2. **Emit SYS_INIT** in oz_dispatch.c that calls +initialize for every class
   that defines it, in topological order (superclass before subclass). Host
   platform uses `__attribute__((constructor))`.
3. **Error on explicit [Class initialize] calls** — transpiler detects these
   and emits a diagnostic error.

## Resolution

Implemented automatic `+initialize` via `OZ_AUTO_INIT` macro:
- **resolve.py**: `_collect_initialize_classes()` gathers classes with `+initialize` in topological order; `_check_initialize_guard()` detects unnecessary Apple-style guard pattern
- **emit.py**: errors on explicit `[Class initialize]` calls; passes `initialize_classes` to dispatch templates
- **oz_dispatch.c.j2**: emits `OZ_AUTO_INIT()` per class
- **PAL**: `SYS_INIT(fn, APPLICATION, 90)` on Zephyr, `__attribute__((constructor))` on host
- 451 transpiler tests, 39 behavior, 12 adapted, 10/10 twister pass
