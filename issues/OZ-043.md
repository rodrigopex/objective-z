## Context

In `samples/hello_category/`, `main.m` accesses a protected ivar directly
from outside the class using the arrow operator. The transpiler generates
valid C code for this, but it violates Objective-C access control semantics.
Ivars in `@interface` are `@protected` by default — only accessible from the
class itself and its subclasses.

## Input

```objc
/* Car.h — ivars are @protected by default */
@interface Car: OZObject {
    struct color *_color;
}
@property(readonly) struct color *color;
@end

/* main.m — external code, NOT a subclass */
Car *myCar = [[Car alloc] initWithColor:&(struct color){255, 255, 0}
                               andModel:@"Honda Civic"];
myCar->_color = &(struct color){1, 2, 3};  /* should be rejected */
```

## Observed

Transpiler emits `myCar->_color = &(struct color){1, 2, 3};` in the
generated C (`main_ozm.c:22`), which compiles as a direct struct member
write. No error or warning is emitted.

## Expected

The transpiler should emit an error when external code accesses a
`@protected` or `@private` ivar via the arrow operator. Only `@public`
ivars should be accessible from outside the class hierarchy.

## Analysis

Three gaps in the transpiler:
1. **model.py** — `OZIvar` has no access control field (no visibility info)
2. **collect.py** — Does not parse ivar access specifiers from `ObjCIvarDecl`
3. **emit.py** — `ObjCIvarRefExpr` handler (lines 1444-1462) blindly emits
   `other->_ivar` when base is not `self`, with no access check

## Workaround

Use property accessors instead of direct ivar access from outside the class.
