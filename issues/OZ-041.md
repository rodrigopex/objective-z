# OZ-041: OZNumber and OZDictionary segfault in behavior tests

- **blocking:** Foundation behavior tests (OZNumber, OZDictionary)
- **assigned-to:** MAINTAINER

- **commit:** 3d97b5b

## Context

Discovered during test coverage work (PR 7: foundation-behavior). OZString and
OZArray behavior tests pass, but OZNumber and OZDictionary tests segfault at
runtime (exit code 245 = signal 11).

## Input

```objc
/* oz-pool: OZObject=1,OZNumber=4 */
#import "OZFoundationBase.h"

@interface NumberTest : OZObject
- (int)boxedIntValue;
@end

@implementation NumberTest
- (int)boxedIntValue {
    OZNumber *n = @(42);
    int v = [n intValue];
    [n release];
    return v;
}
@end
```

## Observed

- Transpilation succeeds (no errors)
- GCC compilation succeeds (no warnings)
- Test binary crashes at runtime: exit code 245 (SIGSEGV)
- No stderr output (crash before Unity can report)

Same behavior for OZDictionary with `@{@"k": @(1)}` literals.

## Expected

The test binary should execute and report Unity test results.

## Analysis

Likely causes (not yet confirmed):
1. Pool dispatch table not correctly initialized for Foundation classes
   when multiple Foundation stubs are merged into one TU
2. Slab pool sizes insufficient for the number of OZNumber/OZDictionary
   allocations (including internal temporaries)
3. vtable entries for OZNumber methods (intValue, boolValue, etc.) not
   populated correctly in oz_dispatch.c when all Foundation classes are
   present

OZString works because string constants are immortal (no alloc/dealloc).
OZArray works because it only stores pre-existing objects. OZNumber and
OZDictionary both require active slab allocation for their instances.

## Workaround

Test OZNumber/OZDictionary indirectly through OZArray behavior tests
(array elements can be boxed numbers).
