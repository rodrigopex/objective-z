## Context

OZ-057 added generic type validation in resolve.py, but it is ineffective in practice. Clang erases ObjC generic type parameters from `qualType` in the JSON AST dump — `OZArray<id<PXDataProcessor>> *` becomes just `OZArray *`. The resolve-time validation only works with hand-crafted test ASTs, not with real Clang output.

Verified by inspecting px-app's `main_m.ast.json`:
```
sensorArray qualType: OZArray *    ← generics stripped by Clang
generic_params: []                  ← nothing to validate
```

## Input

```objc
OZArray<id<PXDataProcessor>> *sensorArray = @[ temp, hum ];
```

## Observed

Clang AST `qualType` field contains `OZArray *` — generic parameters are erased. The OZ-057 validation in resolve.py sees no generic params and skips validation entirely.

## Expected

The transpiler extracts generic type annotations from the original `.m` source via tree-sitter (already a dependency), associates them with VarDecl nodes, and feeds them into the OZ-057 validation pass. A real `.m` test file must be included that goes through the full pipeline (Clang AST → oz_transpile → error) to prove the feature works end-to-end.

## Workaround

None. Generic type mismatches are silently accepted.
