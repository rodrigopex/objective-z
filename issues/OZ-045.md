## Context

In `samples/hello_category/`, accessing `myCar.color` crashes with
`ASSERTION FAIL [z_spin_lock_valid(l)]` because the generated atomic
getter declares a local `oz_spinlock_t lck;` without initializing it,
then passes it to `OZ_SPINLOCK()`.

## Input

```objc
/* Car.h */
@interface Car: OZObject {
    struct color *_color;
}
@property(readonly) struct color *color;  /* atomic by default */
@end
```

## Observed

Generated getter in `Car+Maintenance_ozm.c`:

```c
struct color * Car_color(struct Car *self)
{
    oz_spinlock_t lck;          /* uninitialized stack variable */
    struct color * val;
    OZ_SPINLOCK(&lck) {         /* crashes: invalid spinlock */
        val = self->_color;
    }
    return val;
}
```

Runtime crash:
```
ASSERTION FAIL [z_spin_lock_valid(l)] @ .../zephyr/include/zephyr/spinlock.h:136
    Invalid spinlock 0x20001854
```

## Expected

The spinlock for atomic property access should be either:
1. A per-object field (e.g. `self->base._oz_lock`) initialized at alloc time, or
2. A properly initialized static/local spinlock

The generated getter should not use garbage stack memory as a spinlock.

## Analysis

The emit pass generates atomic getters with a local `oz_spinlock_t lck;`
that is never initialized. The `OZ_SPINLOCK` macro expects a valid,
initialized spinlock. Same pattern affects `Car_model()` and any other
atomic property getter.

## Workaround

Declare properties as `nonatomic` to avoid the spinlock path:
```objc
@property(readonly, nonatomic) struct color *color;
```

## Discussion

## Resolution

Fixed in commit c9fe84d. Atomic property getters now use the per-object
spinlock (`self->base._oz_lock`) instead of an uninitialized local
variable, eliminating the runtime crash.
