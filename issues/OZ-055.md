## Context

Building any sample via `objz_transpile_sources()` produces a `compile_commands.json`
with duplicate entries and non-canonical paths. Additionally, the generated `.clangd`
config does not strip ARM/RISC-V cross-compilation flags, causing clangd to fail on
the host.

## Observed

1. `compile_commands.json` contains 8 duplicate `.m` entries per sample — one canonical
   (`/objective-z/src/OZObject.m`) and one non-canonical
   (`/samples/hello_category/../../src/OZObject.m`). Root cause:
   `ZEPHYR_OBJZ_MODULE_DIR` inherits a non-canonical path from `ZEPHYR_EXTRA_MODULES`
   (`${CMAKE_CURRENT_SOURCE_DIR}/../../`). `file(GLOB_RECURSE)` in
   `_objz_write_compile_db` preserves this prefix; `list(FIND)` fails to match
   against the canonical paths stored by `_objz_collect_compile_db`.

2. `.clangd` Remove list strips `--target=*` but not `-mcpu=*`, `-mthumb`, `-mfpu=*`,
   `-mfloat-abi=*`, `-march=*`, `-mabi=*`, `-mno-relax`. Without a target, clangd
   defaults to the host, but these ARM/RISC-V flags are invalid for the host target,
   causing clangd to reject the entire compilation command.

## Expected

- Zero duplicate entries in `compile_commands.json`
- All paths canonical (no `../` segments)
- clangd parses `.m` files without errors (`clangd --check` = 0 errors)
