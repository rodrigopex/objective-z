## Context

In `px-app/src/main.m` line 72, an `OZArray<id<PXDataProcessor>>` is initialized with `PXTemperatureSensor` and `PXHumiditySensor` objects, which conform to `PXSensorProtocol` — not `PXDataProcessor`. The transpiler silently accepts this because `model.py:_strip_qualifiers()` erases all generic type parameters via regex before any validation occurs.

## Input

```objc
PXTemperatureSensor *temp = [[PXTemperatureSensor alloc] initWithId:1];
PXHumiditySensor *hum = [[PXHumiditySensor alloc] initWithId:2];
OZArray<id<PXDataProcessor>> *sensorArray = @[ temp, hum ];
```

## Observed

Transpiler emits `struct OZArray *` with no diagnostic. The generic type parameter `id<PXDataProcessor>` is silently discarded. No warning that `PXTemperatureSensor` / `PXHumiditySensor` do not conform to `PXDataProcessor`.

## Expected

Emit an error when objects assigned to a generic-typed variable don't conform to the declared generic type constraint. Applies to all generic containers (OZArray, OZDictionary, and any user-defined generic class), not just arrays. This is always an error, no `--strict` gating. Preserve generic type parameter metadata in `OZType` / `OZModule` so `resolve.py` can validate conformance.

## Workaround

None needed — code compiles and runs correctly due to duck typing in generated C. The risk is silent type-safety violations that would be caught by Clang in standard ObjC.
